# CDD 사용자 안내

CDD는 Codex가 승인된 문서 기준으로만 작업하도록 묶는 Codex skill이다.

사람은 보통 이 문서만 읽으면 된다. `SKILL.md`는 Codex가 skill을 발견하기 위한 metadata이고, `_`로 시작하는 파일들은 Codex가 내부 chain에서 읽는 규칙 파일이다.

## 핵심 용어

- 제품 기준 준비 상태: 무엇을 왜 만들 것인지에 대한 기획 준비도.
- 기술 설계 준비 상태: 제품 판단을 저장 구조, 상태, API, 코드 구조로 표현할 기준이 정해졌는지에 대한 설계 준비도.
- 구현 시작 가능 여부: 에이전트가 구현을 시작해도 되는지에 대한 실행 준비도.
- 이번 작업 기준 묶음: 이번 작업에서 따라야 할 승인된 기준 문서 묶음.
- 작업 기준서: 구현 전 작업 범위, 금지 범위, 검증 기준을 고정하는 작업 계약.
- 상호작용 방식 확인: 사용자 또는 운영자가 접하는 기능의 입력, 출력, 흐름, 실패와 피드백을 먼저 확인하는 절차.
- 저장 의미 확인: table, column, migration, repository, API DTO를 말하기 전에 무엇을 왜 저장하는지 확인하는 절차.
- 동작 계약 확인: API path나 request/response를 말하기 전에 어떤 행동과 결과를 약속하는지 확인하는 절차.
- 상태 의미 확인: status enum이나 상태 전이를 말하기 전에 각 상태가 무엇을 뜻하는지 확인하는 절차.
- 운영/품질 기준 확인: 성능, 보안, 권한, 조회, 재시도, 로그/감사, 운영 기준을 구현 전에 확인하는 절차.

## 한 줄 요약

- 보통 작업은 `start-here.md`에서 시작한다.
- 삭제/폐기/정리 작업은 `cleanup-delete.md`에서 시작한다.
- 요청이 설명, 설계, 문서 수정, 구현, 삭제, 검증 중 무엇인지 애매하면 먼저 질문한다.
- 구현은 제품 기준 문서, 기술 설계 기준 문서, 이번 작업 기준 묶음, 준비 상태 확인이 준비된 뒤에만 진행한다.
- 사용자 또는 운영자가 접하는 기능에서는 상호작용 방식도 제품 판단이다.
- table, column, API path, status enum은 제품 판단의 결과다. CDD는 이것들이 제품 판단을 대신하게 두지 않는다.
- 성능, 보안, 권한, 데이터 양, 조회 방식, 실패 처리, 재시도, 로그/감사 같은 운영/품질 기준도 설계 판단이다.
- CDD 문서는 제품 기준 문서가 아니다. 제품/엔지니어링 기준 문서는 대상 프로젝트 저장소에 둔다.

## 사용자 보고 방식

CDD는 내부적으로 엄격하게 판단하되, 사용자가 보는 기본 보고에서는 내부 용어를 먼저 노출하지 않는다.

- `SOT` 대신 "확인한 기준 문서" 또는 "현재 기준"이라고 쓴다.
- `Product Readiness` 대신 "제품 방향"이라고 쓴다.
- `Engineering Readiness` 대신 "설계 준비 상태"라고 쓴다.
- `Implementation Readiness` 대신 "지금 바로 만들 수 있는지"라고 쓴다.
- `Vertical Slice` 대신 "첫 기능 범위"라고 쓴다.
- `DB/API/UI 결정` 대신 "저장 방식 / 동작 방식 / 화면 흐름"이라고 쓴다.
- `Storage Intent Check` 대신 "무엇을 왜 저장할지"라고 쓴다.
- `Behavior Contract Check` 대신 "사용자가 어떤 행동을 하고 어떤 결과를 받는지"라고 쓴다.
- `State Meaning Check` 대신 "상태값이 무엇을 의미하는지"라고 쓴다.
- `READY` / `NOT READY` 대신 "바로 진행 가능" / "아직 결정 필요"라고 쓴다.

기본 보고는 다음 형식을 우선한다.

```text
확인한 기준:
- ...

현재 판단:
- 바로 구현 가능 / 아직 결정 필요

이유:
- ...

먼저 정할 것:
1. ...
2. ...

내 추천:
- ...

다음에 할 일:
- ...

내가 물어볼 것:
1. ...
2. ...
```

사용자가 "CDD 판정표", "내부 판정", "상세 harness status"를 요청했거나 구현 지시서, 작업 기준서, 검증 산출물 같은 에이전트 간 전달물을 작성할 때만 내부 용어와 enum을 그대로 쓸 수 있다.

## 다음 행동 원칙

CDD는 단순히 "불가", "부족", "완료"로 끝내지 않는다. 모든 사용자-facing 응답은 마지막에 "다음에 할 일"을 포함한다.

### 사용자 선택이 필요한 경우

```text
다음에 할 일:
아직 직접 진행하면 안 됩니다. 먼저 아래 중 하나를 선택해야 합니다.

선택지:
1. ...
2. ...
3. ...

제 추천:
- ...

바로 답할 수 있는 문장:
"..."
```

### 사용자 개입 없이 진행 가능한 경우

```text
다음에 할 일:
사용자 선택이 필요한 부분은 없습니다.
현재 기준으로 안전하게 진행할 수 있으므로, 요청 범위 안에서 다음 작업까지 진행합니다.

진행할 작업:
- ...

진행하지 않을 작업:
- ...
```

이 경우 CDD는 보고만 하고 멈추지 않고, 실제로 문서 보강, 구현, 검증, 보고 등 허용된 다음 단계까지 진행해야 한다.

### 완료한 경우

```text
다음에 할 일:
이번 작업은 완료되었습니다.

다음 후보:
1. ...
2. ...
3. ...

제 추천:
- ...
```

## 자동 진행과 중단

자동 진행 가능한 경우:

- 사용자의 요청이 명확하다.
- 파일 수정이 허용되어 있다.
- 제품 기준과 설계 기준이 충분하다.
- 저장, 동작, 상태, 상호작용, 운영 기준이 충분하다.
- 작업 범위가 작고 명확하다.
- 금지된 작업에 닿지 않는다.
- 데이터 손실, public API 제거, migration 위험, dependency 대량 변경이 없다.
- 검증 방법이 명확하다.

자동 진행하면 안 되는 경우:

- 제품 방향이 비어 있다.
- 사용자/운영자 상호작용 방식이 비어 있다.
- 무엇을 왜 저장할지 비어 있다.
- 어떤 행동과 결과를 제공할지 비어 있다.
- 상태값 의미가 비어 있다.
- 성능/보안/운영 기준이 비어 있다.
- 삭제와 보존 중 선택이 필요하다.
- migration, 데이터 삭제, public API 제거가 있다.
- dependency 변경 영향이 크다.
- 사용자의 요청이 분석/설계/문서 수정/구현 중 무엇인지 애매하다.
- 사용자가 "아직 수정하지 마라"고 했다.

정책이나 설계가 비어 있으면 CDD는 임의로 채우지 않고 사용자에게 선택을 요구한다.

## 설치

이미 등록했다면 이 단계는 건너뛰어도 된다.

```sh
CDD_ROOT=/path/to/cdd
mkdir -p ~/.agents/skills
ln -s "$CDD_ROOT" ~/.agents/skills/cdd
```

정상 구조는 다음과 같다.

```text
~/.agents/skills/cdd/SKILL.md
```

등록 후에는 Codex 앱/세션을 다시 시작해야 새 skill이 잡힐 수 있다.

Codex app UI용 metadata가 필요하면 `agents/openai.yaml`을 함께 둔다. 현재 CDD는 icon이나 asset 없이 표시 이름과 짧은 설명만 제공한다.

## 바로 쓰는 방법

새 작업을 시작할 때:

```text
CDD 스킬로 이 작업을 시작해줘.
start-here.md에서 시작하고, 아직 구현하지 마.
먼저 확인한 기준, 현재 판단, 바로 구현하면 안 되는 이유를 보고하고,
부족한 결정이 있으면 사용자가 답해야 할 질문으로 정리해줘.
```

요청이 애매할 때 CDD는 바로 구현하거나 파일을 수정하지 않고 이렇게 물어야 한다.

```text
지금 요청이 실제 파일 수정을 원하는 것인지, 먼저 설계/계획만 보려는 것인지 확인이 필요합니다.
어느 쪽으로 진행할까요?
1. 현재 상태를 읽고 판단만 한다
2. 문서 초안을 만든다
3. 문서를 실제로 수정한다
4. 구현 계획만 만든다
5. 코드를 실제로 수정한다
6. 삭제/정리 대상을 먼저 목록화한다
```

삭제/폐기/정리 작업일 때:

```text
CDD 스킬로 cleanup/delete 작업을 진행해줘.
cleanup-delete.md에서 시작하고, keep/delete list와 승인 지점이 분명해질 때까지 파일을 수정하지 마.
```

검증만 요청할 때:

```text
CDD 스킬로 결과를 검증해줘.
verify-work.md에서 시작하고, 제품 기준 문서와 기술 설계 기준 문서, 이번 작업 기준 묶음, 작업 기준서에 어긋난 점을 찾아줘.
```

완료 보고를 요청할 때:

```text
CDD 스킬로 완료 조건을 확인해줘.
complete-work.md 기준으로 검증 결과, 금지사항 위반 여부, 남은 위험만 짧게 보고해줘.
```

## 사람이 준비할 것

모든 것을 처음부터 완성해 둘 필요는 없다. 부족하면 Codex가 아직 필요한 결정을 질문해야 한다.

- 제품 기준 문서: 무엇을 왜 만들지, 사용자와 범위, 하지 않을 것, 성공 기준.
- 상호작용 방식: 사용자 또는 운영자가 어디서 시작하는지, 무엇을 입력하는지, 무엇을 출력으로 받는지, 실패와 빈 상태와 처리 중 피드백을 어떻게 받을지.
- 기술 설계 기준 문서: 아키텍처, 도메인 모델, 데이터 흐름, API/DB/상태/권한/테스트/예외 처리 판단.
- 저장 의미: 무엇을 저장하는지, 왜 저장하는지, 나중에 어떻게 다시 읽는지, 무엇은 저장하지 않는지, 구조화할 것과 자유 텍스트로 둘 것.
- 동작 계약: 사용자가 기대하는 동작, 입력, 성공/실패 결과, 권한, 재시도/멱등성, 외부에 노출할 표현.
- 상태 의미: 각 상태의 뜻, 상태를 바꾸는 사건, 상태별 허용 행동, 저장할지 계산할지, 외부에 노출할지.
- 운영/품질 기준: 데이터 양, 조회 방식, 정렬/검색/페이지 처리, 응답 속도, 권한 검증, 민감 정보 노출, 재시도, 로그/감사.
- 이번 작업 기준 묶음: 이번 작업에서 실제로 따라야 하는 승인된 기준 묶음.
- 승인 범위: 수정해도 되는 파일과 수정하면 안 되는 파일.
- 검증 명령: 이번 작업이 끝났을 때 실행해야 하는 확인 방법.

## Codex가 먼저 보고해야 하는 것

작업을 바로 구현하면 안 된다. 시작 전에 Codex는 최소한 다음을 먼저 말해야 한다.

```text
확인한 기준:
- ...

현재 판단:
- 바로 구현 가능 / 아직 결정 필요

이유:
- ...

먼저 정할 것:
1. ...
2. ...

내 추천:
- ...

다음에 할 일:
- ...

내가 물어볼 것:
1. ...
2. ...
```

현재 판단이 "아직 결정 필요"이면 구현하지 않고 부족한 결정을 질문한다. 내부 산출물에서는 필요하면 `READY` / `NOT_READY` 같은 enum을 기록할 수 있지만, 기본 사용자 보고에서는 자연어로 번역한다.

사용자 또는 운영자가 접하는 기능이면 Codex는 인터페이스를 먼저 만들지 않고 다음을 먼저 확인해야 한다.

```text
상호작용 방식 확인:
- 이 기능을 사용하는 사람은 누구인가?
- 사용자는 어디서 이 기능을 발견하거나 실행하는가?
- 사용자는 무엇을 입력하는가?
- 성공하면 무엇을 보여주거나 반환하는가?
- 실패하면 무엇을 보여주거나 반환하는가?
- 데이터가 없을 때 무엇을 보여주거나 반환하는가?
- 권한이 없거나 접근할 수 없을 때 어떻게 반응하는가?
- 처리 중이거나 대기 중일 때 어떤 피드백을 주는가?
- 아직 정하지 못한 상호작용 결정:
- 결론: 상호작용 설계 가능 / 상호작용 설계 보류
```

DB/API/status가 포함된 작업이면 Codex는 구조를 먼저 제안하지 않고 다음을 먼저 확인해야 한다. 기본 사용자 보고에서는 내부 체크 이름 대신 자연어 제목을 쓴다.

```text
무엇을 왜 저장할지:
- 저장하려는 것:
- 저장하는 이유:
- 나중에 읽는 방식:
- 저장하지 않을 것:
- 구조화할 것:
- 자유 텍스트로 둘 것:
- 수정/삭제/보존 정책:
- 소유권/범위:
- 부족한 결정:
- 결론: 저장 방식 설계 가능 / 저장 방식 설계 보류
```

저장 방식 설계가 보류 상태이면 table, column, migration, repository, API DTO를 제안하지 않고 아직 필요한 결정을 질문한다. 동작 계약이 없으면 API path를 제안하지 않고, 상태 의미가 없으면 status enum을 제안하지 않는다. 내부 artifact에서는 필요하면 `DB_DESIGN_ALLOWED` / `DB_DESIGN_BLOCKED` 같은 enum을 기록할 수 있다.

성능, 보안, 운영 기준이 필요한 작업이면 다음을 먼저 확인해야 한다.

```text
운영/품질 기준 확인:
- 예상 데이터 양은 어느 정도인가?
- 목록이나 조회가 있다면 정렬, 검색, 페이지 처리가 필요한가?
- 사용자가 기대하는 응답 속도는 어느 정도인가?
- 권한 검증은 어디에서 필요한가?
- 민감 정보나 외부 연동 정보가 노출될 위험이 있는가?
- 실패 시 사용자는 무엇을 알 수 있어야 하는가?
- 재시도, 중복 실행 방지, 멱등성이 필요한가?
- 로그나 감사 기록이 필요한가?
- 아직 정하지 못한 운영/품질 결정:
- 결론: 설계 가능 / 설계 보류
```

`설계 보류`이면 안전하다고 가정하지 않고 먼저 아직 필요한 결정을 질문한다.

## Public Entry Points

사용자가 직접 호출해도 되는 파일이다.

- `start-here.md`: 일반 작업 시작점.
- `plan-task.md`: 제품/기술 기준이 준비된 뒤 Plan/Task를 잡을 때.
- `write-implementation-prompt.md`: 승인된 Task를 구현 지시서로 바꿀 때.
- `cleanup-delete.md`: 삭제, 폐기, 정리 작업.
- `verify-work.md`: 결과가 기준 문서와 작업 기준서에 맞는지 검증할 때.
- `revise-work.md`: 검증 실패 후 수정 지시서를 만들 때.
- `complete-work.md`: 짧은 완료 보고를 만들 때.

Public entrypoint도 그 자체로 파일 수정 권한을 주지 않는다. 권한은 작업 방식, 이번 작업 기준 묶음, 승인 지점이 결정한다.

## Internal Modules

`_`로 시작하는 파일은 사람이 직접 호출하지 않는다. Codex가 public entrypoint 흐름 안에서만 읽는다.

- `_work-mode.md`
- `_authority-boundary.md`
- `_project-context.md`
- `_source-of-truth-manager.md`
- `_document-readiness.md`
- `_missing-context.md`
- `_document-supplement.md`
- `_readiness-gates.md`
- `_sot-packet.md`
- `_implementation-rules.md`
- `_artifact-metadata.md`
- `_artifact-templates.md`
- `_status-machine.md`
- `_approval-reference.md`
- `_user-facing-language.md`

Internal module은 단독으로 구현, 삭제, 기준 문서 변경, completion을 승인할 수 없다. 단독으로 `IMPLEMENTATION_ALLOWED`, `APPLY_AUTHORIZED`, `PATCH_AUTHORIZED`, `COMPLETE`도 만들 수 없다.

## 흐름

```text
사용자 요청
-> start-here.md 또는 cleanup-delete.md
-> 작업 방식 / 이번 작업 기준 / 준비 상태 확인
-> 아직 필요한 결정 질문 또는 Plan/Task
-> 구현 지시서 작성
-> 사용자 승인
-> 구현
-> 검증
-> 필요하면 revision
-> 완료 보고
```

## 하지 말아야 할 것

- `_*.md` 파일을 직접 entrypoint처럼 호출하지 않는다.
- CDD 문서를 제품 기준 문서나 기술 설계 기준 문서로 쓰지 않는다.
- "좋아", "진행해" 같은 말만으로 구현 승인이나 APPLY 승인으로 해석하지 않는다.
- "좋아", "진행해", "다음", "반영해" 같은 말만으로 파일 수정 승인, 문서 수정 승인, 삭제 승인으로 해석하지 않는다.
- 애매한 요청을 구현 요청으로 승격하지 않는다.
- 애매한 요청을 파일 수정 승인으로 해석하지 않는다.
- 사용자의 의도를 추측해서 파일을 수정하지 않는다.
- 준비 상태가 부족한데 먼저 구현하지 않는다.
- 사용자/운영자 상호작용이 비어 있는데 화면, CLI 명령, API surface, 배치 실행 방식을 제안하지 않는다.
- 무엇을 왜 저장할지 정해지기 전에 table, column, migration, repository, API DTO를 제안하지 않는다.
- 동작 계약이 정해지기 전에 API path나 request/response shape를 제안하지 않는다.
- 상태 의미가 정해지기 전에 status enum이나 상태 전이를 제안하지 않는다.
- 운영/품질 기준이 비어 있는데 성능, 보안, 권한, 조회, 재시도, 로그/감사를 안전하다고 가정하지 않는다.
- cleanup/delete를 새 기능 구현이나 리팩토링으로 확장하지 않는다.
- CLI, 스크립트, 자동 수정 플랫폼을 CDD 본체에 추가하지 않는다.

## 자세한 규칙을 고쳐야 할 때

CDD 자체를 유지보수할 때만 internal module을 읽는다. 사용자가 internal module을 직접 언급해도 Codex는 가장 가까운 public entrypoint로 라우팅해야 한다.
