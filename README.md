# CDD

CDD는 AI 작업 에이전트가 구현 전에 필요한 기준을 확인하게 하는 문서 기반 작업 하네스다.

보통 사용자는 이 README를 자세히 읽을 필요가 없다. 프로젝트 작업 요청에 CDD 링크나 `$cdd`를 함께 넣으면 에이전트가 `SKILL.md`와 public entrypoint를 따라 작업을 시작한다.

## 쓰는 방법

예:

```text
$cdd
이 작업을 구현해줘.
```

```text
CDD 기준으로 먼저 판단해줘.
아직 파일은 수정하지 마.
```

```text
$cdd
검증 결과를 보고 완료 가능한지 확인해줘.
```

## CDD가 막는 것

- 요청이 애매한데 구현이나 파일 수정을 시작하는 것
- 제품 방향이나 사용자/운영자 상호작용이 비어 있는데 화면, API, CLI, 배치 실행 방식을 만드는 것
- 무엇을 왜 저장할지 정해지기 전에 table, column, migration, repository, API DTO를 제안하는 것
- 어떤 행동과 결과를 제공할지 정해지기 전에 API path나 request/response를 제안하는 것
- 상태 의미가 정해지기 전에 status enum이나 상태 전이를 제안하는 것
- 성능, 보안, 권한, 조회, 실패 처리, 재시도, 로그/감사 기준이 비어 있는데 안전하다고 가정하는 것
- 대상 프로젝트의 기존 문서 구조를 확인하지 않고 작업 기준서, 구현 지시서, 검증 결과, 완료 기록 파일을 새로 흩어 만드는 것

## CDD가 진행하는 것

기준이 충분하고 위험 변경이 없으면 CDD는 다시 묻지 않고 요청 범위 안에서 다음 단계까지 이어간다.

예:

- 문서 보강 요청이면 문서 보강 후 검증까지 진행
- 구현 요청이고 기준이 충분하면 구현, 검증, 보고까지 진행
- 작업 지시서가 내부 실행 기준일 뿐이면 초안 승인과 실행 승인을 기계적으로 반복하지 않음
- 선행 작업이 검증 완료 상태이고 남은 것이 완료 기록 정합성뿐이면 정리 후 후속 작업으로 이어감

## CDD가 멈추는 경우

- 사용자가 원하는 작업이 설명, 설계, 문서 수정, 구현, 삭제, 검증 중 무엇인지 불명확함
- 제품 판단이나 설계 판단이 비어 있음
- 저장 의미, 동작 계약, 상태 의미, 상호작용 방식, 운영/품질 기준이 부족함
- 삭제/보존 선택, migration, 데이터 삭제, public API 제거, 큰 dependency 변경이 있음
- 기존 문서 구조와 다른 파일 배치를 하려 함
- 사용자가 "분석만", "수정하지 마", "초안만", "먼저 보여줘"처럼 제한을 둠

멈출 때는 내부 상태명보다 자연어로 이유와 선택지를 말해야 한다.

## 문서 구조

사용자가 직접 호출할 수 있는 public entrypoint:

- `start-here.md`
- `plan-task.md`
- `write-implementation-prompt.md`
- `cleanup-delete.md`
- `verify-work.md`
- `revise-work.md`
- `complete-work.md`

`_`로 시작하는 파일은 내부 chain module이다. 사용자가 직접 호출하기보다 public entrypoint 흐름 안에서 에이전트가 읽는다.

## 설치

이미 등록했다면 건너뛰어도 된다.

```sh
CDD_ROOT=/path/to/cdd
mkdir -p ~/.agents/skills
ln -s "$CDD_ROOT" ~/.agents/skills/cdd
```

정상 구조:

```text
~/.agents/skills/cdd/SKILL.md
```

등록 후에는 사용하는 에이전트 앱이나 세션을 다시 시작해야 새 skill이 잡힐 수 있다.

## 유지보수

- skill 인식 기준은 `SKILL.md`에 둔다.
- 사용자가 직접 부르는 흐름은 public entrypoint에 둔다.
- 세부 판정 규칙은 `_*.md` 내부 module에 둔다.
- README는 링크로 들어온 사람이 빠르게 이해하는 짧은 안내서로 유지한다.
