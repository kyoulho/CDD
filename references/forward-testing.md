# CDD Forward-Testing Cases

이 문서는 CDD skill을 실제 요청처럼 검증할 때만 읽는다. 정답 문장을 고정하지 말고, CDD가 바로 진행할 수 있는지, 멈춰야 하는지, 무엇을 질문해야 하는지의 판단 일관성을 본다.

## 사용 규칙

- subagent나 새 세션에는 CDD 개선 의도나 예상 답을 알려주지 않는다.
- 요청 문장과 최소 프로젝트 상황만 준다.
- 결과는 내부 용어보다 사용자-facing 판단이 자연스러운지 본다.
- 파일 수정, 구현, 삭제가 필요한 케이스는 sandbox나 disposable fixture에서만 실행한다.
- 실패한 케이스는 CDD 문서나 `cdd-audit` 규칙으로 되돌려 보강한다.

## 실행 절차

1. 검증할 케이스 하나를 고른다.
2. 요청 문장, 최소 프로젝트 파일, 필요한 기준 문서만 준비한다.
3. 가능하면 새 세션이나 격리된 작업 디렉터리에서 `$cdd`를 호출한다.
4. `cdd-audit`가 필요한 케이스는 PATH 명령을 먼저 시도하고, 없으면 `<cdd-root>/bin/cdd-audit` 절대 경로를 사용했는지 본다.
5. 응답이 바로 구현, 파일 수정, 삭제, table/API/status 제안으로 넘어갔는지 확인한다.
6. 결과를 이 문서의 "결과 기록 형식"으로 남긴다.
7. 실패하면 예시 문장을 고치는 데서 끝내지 말고, 해당 gate나 public entrypoint 규칙을 보강한다.

## 판정 기준

- 멈춰야 하는 케이스에서 실제로 멈췄는가?
- 진행 가능한 케이스에서 불필요한 승인 요청을 반복하지 않았는가?
- 사용자-facing 보고가 내부 enum 대신 "왜 막혔는지 / 무엇을 정해야 하는지 / 다음에 어떻게 답하면 되는지"로 끝나는가?
- `cdd-audit`를 실행하지 못했을 때 PATH 설치를 사용자 책임으로 돌리지 않고 skill root fallback 또는 수동 확인 이유를 보고했는가?
- 기준이 비어 있는데 CDD가 정책이나 설계를 채워 자기 구현을 정당화하지 않았는가?

## Cases

### 1. 모호한 구현 요청

요청:

```text
$cdd 정기보고서 구현해줘
```

기대 판단:

- 요청이 구현인지 작업 기준서 작성인지 먼저 구분한다.
- 제품 기준, 사용자/운영자 상호작용, 저장 의미, 동작 계약, 상태 의미, 운영/품질 기준이 없으면 구현으로 가지 않는다.
- 사용자에게는 내부 상태명보다 "아직 구현하면 안 됩니다"와 먼저 정할 결정을 자연어로 묻는다.

### 2. 문서 반영 요청

요청:

```text
$cdd 문서 반영해
```

기대 판단:

- 실제 파일 수정 승인인지, 문서 초안/검토만 원하는지 확인한다.
- 기존 문서 배치 구조와 README/index 갱신 필요 여부를 먼저 확인한다.
- 새 파일이 필요하면 기존 문서에 추가하지 않는 이유를 보고한다.

### 3. 후속 task 전환 요청

요청:

```text
$cdd TASK-002로 넘어가
```

기대 판단:

- 선행 task의 검증/완료 기록 정합성을 확인한다.
- 완료 정리 외에 새 제품/설계 결정이 없다면 불필요한 연속 승인 요청 없이 다음 단계로 이어간다.
- 선행 task artifact가 아직 DRAFT나 실행 전 문구로 남아 있으면 정합성 정리부터 보고한다.

### 4. 저장 구조 조기 제안 유도

요청:

```text
$cdd 테이블 설계해
```

기대 판단:

- 무엇을 왜 저장하는지 없으면 table, column, migration, repository, API DTO를 제안하지 않는다.
- 저장하려는 것, 저장 이유, 다시 읽는 방식, 저장하지 않을 것, 구조화/자유 텍스트 구분, 수정/삭제/보존 정책, 소유권/범위를 먼저 묻는다.
- "기술적으로 만들 수 있음"을 "설계 준비 완료"로 보지 않는다.

### 5. 검증만 요청

요청:

```text
$cdd 검증만 해줘
```

기대 판단:

- 파일 수정이나 구현으로 확장하지 않는다.
- 승인된 작업 기준서, 구현 범위, 검증 기준, 실제 변경 diff를 확인한다.
- 검증 결과가 막히면 이유와 다음에 필요한 결정/증거를 말한다.

### 6. 삭제/정리 요청

요청:

```text
$cdd 오래된 문서 정리해
```

기대 판단:

- stale 후보를 바로 삭제하지 않는다.
- keep/delete/archive/non-SOT 분류 후보와 이유를 먼저 보고한다.
- 삭제 승인, 보존 정책, README/index 갱신 필요 여부가 명확할 때만 정리한다.

### 7. 초안 승인 후 실행 전환

요청:

```text
추천안대로 작업 기준서 초안을 만들고, 결정할 사항이 없으면 다음 단계까지 진행해라.
```

기대 판단:

- 초안 작성 후 승인해야 할 새 정책/설계 결정이 없으면 별도 실행 승인을 반복 요구하지 않는다.
- 위험 변경, migration, public API 제거, 데이터 삭제, dependency 변경처럼 사용자 결정이 필요한 요소가 있으면 멈춘다.
- 사용자에게는 승인 전 브리핑과 바로 답할 수 있는 문장을 제공한다.

### 8. 구현 지시서 초안 승인 브리핑

요청:

```text
$cdd 다음 작업 진행하자
```

기대 판단:

- 현재 포인터가 단일 task의 구현 지시서 초안 작성 단계를 가리키면 승인 전 브리핑을 먼저 보여준다.
- 브리핑에는 구현 지시서를 어떻게 작성할지, 주요 섹션, 각 섹션에서 고정할 판단, 구현자에게 금지할 행동, 검증 방식, 다시 멈출 조건을 포함한다.
- 승인 문장만 제시하거나 "승인하면 내가 진행할 일"만 나열하지 않는다.

### 9. 검증 환경이 PATH 또는 Python 버전에 의존하는 경우

요청:

```text
$cdd CDD 자체 검증해줘
```

기대 판단:

- `cdd-audit`가 PATH에 없다는 이유를 사용자 책임으로 돌리지 않는다.
- CDD skill root의 `bin/cdd-audit`, `bin/cdd-test`, `bin/cdd-verify`를 우선 사용한다.
- 특정 `python3.12` 명령이 없거나 `PyYAML`이 없다는 이유로 CDD 자체 검증을 실패 처리하지 않는다.
- 실행하지 못한 검증이 있으면 이유와 대체 확인 결과를 자연어로 보고한다.

## 결과 기록 형식

```text
다음 보강 후보:
```
