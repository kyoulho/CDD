# Approval Briefing Language

> Access: Internal chain module.
> 내부용 chain module이다. task entrypoint로 직접 호출하지 마라.
> 이 module만으로는 implementation, SOT changes, cleanup/delete, completion을 승인할 수 없다.

이 문서는 사용자 승인 요청 전에 보여줄 브리핑 형식을 정의한다. 브리핑은 승인 문장의 장식이 아니라 사용자가 승인 여부를 판단하는 검토 표면이다.

## 빠른 탐색

- 모든 승인 요청은 "승인 전 브리핑 형식"을 먼저 본다.
- 선행 작업 완료 뒤 다음 task 승인으로 넘어갈 때는 "후속 작업 승인 요청 브리핑"을 본다.
- 사용자-facing 표현 전체 규칙은 `_user-facing-language.md`를 본다.
- approval 종류와 record 기준은 `_approval-reference.md`를 본다.

## 승인 전 브리핑 형식

작업 기준서, 구현 지시서, 기준 문서 변경안, 검증 결과, 완료 기록, 수정 지시서, cleanup/delete 분류안처럼 사용자 승인이 필요한 산출물은 승인 문장을 보여주기 전에 반드시 사용자-facing 브리핑을 먼저 출력한다. 브리핑 없이 승인 문장만 제시하지 않는다. 단순 상태 조회나 다음 작업 조회는 승인 요청이 아니므로 `_user-facing-language.md`의 "조회형 질문 응답 형식"을 사용한다.

브리핑 없이 승인 요청 금지. 첫 번째 승인 요청이든, 두 번째 승인 요청이든, 연속 승인 요청이든, 결합 승인 요청이든 예외는 없다. 몇 번째 승인 요청이든 사용자가 승인 여부를 판단할 수 있는 브리핑이 먼저 있어야 하며, 승인 문장은 브리핑 뒤에만 둔다.

브리핑은 단순한 친절한 설명이 아니라 사용자가 승인 여부를 판단하는 검토 표면이다. 에이전트도 사람도 과거 task 히스토리와 현재 기준을 모두 기억하지 못할 수 있으므로, 브리핑만 읽어도 "무엇을 승인하는지", "무엇은 아직 승인하지 않는지", "승인하면 어떤 결정이 고정되는지"를 알 수 있어야 한다.

완료 기록 정리 뒤 후속 작업으로 넘어가며 사용자에게 "다음에 실제로 진행할 문장"을 제시하는 경우도 승인 요청이다. 후속 작업의 작업 기준서 작성, 구현 지시서 초안 작성, 구현 지시서 실행, 검증 결과 승인, 완료 기록 승인 문장은 모두 브리핑 뒤에 둔다.

사용자가 "다음 작업 진행하자", "다음 단계로 가자", "진행하자"처럼 실제 다음 단계 수행을 요청했고, CDD가 바로 실행하지 못해 특정 승인 문장을 요구해야 한다면 조회형 응답이 아니다. 이 경우 반드시 아래 브리핑 형식을 사용한다.

현재 작업 포인터가 단일 다음 task와 다음 승인 단계를 명확히 가리키면, 일반 선택지 목록을 먼저 제시하지 않는다. 추천 선택지가 사실상 하나라면 해당 승인 대상 브리핑을 바로 보여준다. 우선순위 변경이나 보류는 "다른 선택도 가능" 정도로 보조적으로 말할 수 있지만, 기본 응답을 선택지 목록으로 대체하지 않는다.

사용자가 같은 다음 작업 요청을 반복해도 브리핑을 승인 문장 중심으로 축약하지 않는다. 반복 요청에서는 "같은 승인 대상입니다"라고 짧게 시작할 수 있지만, 현재 기준, 포함되는 것, 제외되는 것, 승인하면 고정되는 결정, 주의할 점, 승인 후 내가 진행할 일, 바로 답할 수 있는 문장은 매번 유지한다.

브리핑은 내부 enum이나 CDD 용어 중심이 아니라 사용자가 무엇을 허용하는지 판단할 수 있는 제품/작업 언어로 작성한다. 너무 길게 쓰지 말고, 승인 여부를 결정하는 데 필요한 허용 범위, 비허용 범위, 위험만 압축한다.

필수 항목:

- 먼저 확인할 정책/작업 결정
- 각 결정에 대한 CDD Agent의 추천
- 현재 기준과 과거 기록 중 이번 승인 판단에 필요한 배경
- 이 승인이 허용하는 행동
- 아직 허용하지 않는 행동
- 승인하면 고정되는 결정
- 사용자가 주의해야 할 위험/중단 조건
- 승인 후 내가 실제로 진행할 일
- 바로 답할 수 있는 승인 문장

후속 작업 승인 요청에서는 아래 항목명이 기본이다.

- 승인 전에 확인할 내용
- 이번 승인의 목적
- 현재 기준
- 포함되는 것
- 제외되는 것
- 승인하면 고정되는 결정
- 주의할 점
- 승인 후 내가 진행할 일
- 바로 답할 수 있는 문장

구현 지시서 초안 작성 승인 요청에서는 위 항목에 더해 "구현 지시서를 어떻게 작성할지"를 함께 설명한다. 사용자는 초안 작성 자체를 승인하기 전에, 승인하면 어떤 목차와 어떤 판단이 문서화되는지 알아야 한다.

이 설명에는 최소한 다음이 있어야 한다.

- 구현 지시서에 담길 주요 섹션
- 각 섹션에서 고정할 판단
- 구현자에게 금지할 행동
- 지시서가 요구할 검증 방식
- 초안 작성 중 다시 멈출 조건

이 중 "승인 후 내가 진행할 일"만 있고 포함/제외/고정되는 결정/주의할 점이 없으면 브리핑이 아니다.

브리핑은 승인 문장을 정당화하는 요약이어야 한다. 승인 문장이 없어도 사용자가 이번 승인을 할지 말지 판단할 수 있을 만큼의 현재 기준, 포함 범위, 제외 범위, 고정되는 결정을 먼저 제시한다.

구현 지시서 브리핑 뒤의 바로 답할 수 있는 문장은 초안 작성만 승인하는지, 실제 실행까지 함께 승인하는지 분명히 구분해야 한다. `구현 지시서 초안 작성을 승인합니다`는 초안 작성만 허용한다. 실제 구현까지 이어가려면 `구현 지시서 초안 작성과 실제 구현을 함께 승인합니다`처럼 실행 대상이 문장 안에 있어야 한다.

권장 형식:

```text
승인 전에 확인할 내용:

이번 승인의 목적:
- ...

먼저 확인할 결정:
1. ...
   제 추천: ...
2. ...
   제 추천: ...

이 승인은 다음을 허용합니다:
- ...

아직 허용하지 않는 것:
- ...

승인하면 이렇게 고정됩니다:
1. ...
2. ...

주의할 점:
- ...

내 추천:
- ...

승인 후 내가 진행할 일:
- ...

구현 지시서는 이렇게 작성합니다:
1. ...
   고정할 판단: ...

이 방향이 맞으면 이렇게 답해 주세요:
"..."
```

금지:

- 브리핑 없이 승인 문장만 출력한다.
- "다음에 실제로 진행할 문장:" 아래 승인 문장만 출력한다.
- "승인하면 내가 진행할 일:" 또는 "승인하면 진행할 일:"만 나열하고 포함/제외/고정되는 결정/주의할 점을 생략한다.
- "CDD 기준상 승인 문구로 승격할 수 없습니다"라고 말한 뒤, 사용자가 무엇을 승인하는지 설명하지 않고 승인 문장만 요구한다.
- 현재 포인터가 단일 다음 task를 가리키는데 "선택지" 목록을 먼저 보여주고 승인 대상 브리핑을 생략한다.
- 반복 요청이라는 이유로 "승인 대상은 하나입니다"와 승인 문장만 남기고 현재 기준, 포함/제외, 고정되는 결정, 주의할 점을 생략한다.
- 내부 상태값, enum, metadata만 나열하고 사용자가 승인 범위를 판단하게 하지 않는다.
- 일반 사용자 보고에서 `PROMPT_DRAFT_APPROVAL`, `PROMPT_EXECUTION_APPROVAL`, `PATCH_APPROVAL`, `APPLY_APPROVAL` 같은 내부 approval enum을 그대로 보여준다.
- 일반 사용자 보고에서 `현재 gate`, `다음 gate`, `gate 통과`, `gate 차단`, `readiness gate` 같은 내부 진행 상태명을 그대로 보여준다. "현재 상태", "다음 단계", "기준 충족", "아직 진행하면 안 됨", "진행 전 확인 기준"으로 바꾼다.
- "문서에 반영했다. 승인 문장은 ..."처럼 승인 전에 사용자가 검토할 내용을 생략한다.
- 승인 전 사용자에게 코드 구현, 문서 적용, 완료 처리가 이미 진행된 것처럼 오해하게 말한다.

## 연속/결합 승인 요청 브리핑

작업 기준서 승인과 구현 지시서 초안 작성 승인을 한 문장으로 묶는 경우에도 두 개의 승인 대상이 있다. 이때 CDD Agent는 결합 승인 문장만 제시하면 안 된다.

반드시 다음 순서로 브리핑한다.

1. 작업 기준서 승인 브리핑
   - 작업 기준서가 무엇을 고정하는지
   - 포함 범위와 제외 범위
   - 승인하면 고정되는 제품/설계/검증 결정
   - 아직 허용하지 않는 작업
2. 구현 지시서 초안 작성 브리핑
   - 구현 지시서를 어떻게 작성할지
   - 구현 지시서에 담길 주요 섹션
   - 각 섹션에서 고정할 판단
   - 구현자에게 금지할 행동
   - 초안 작성 중 다시 멈출 조건
3. 바로 답할 수 있는 결합 승인 문장

나쁜 예:

```text
다음 승인 문장:
"TASK-017 작업 기준서를 승인하고, TASK-017 구현 지시서 초안 작성을 승인합니다."
```

좋은 예:

```text
승인 전에 확인할 내용:

이번 승인은 두 가지를 함께 허용합니다.
1. TASK-017 작업 기준서 승인
2. TASK-017 구현 지시서 초안 작성

작업 기준서 승인 브리핑:
- 이번 작업 기준서는 [무엇]을 첫 구현 범위로 고정합니다.
- 제외되는 것은 [무엇]입니다.
- 승인하면 [검증 기준]이 완료 판단 기준이 됩니다.

구현 지시서 초안 작성 브리핑:
- 구현 지시서는 작업 목표, 구현 범위, 금지 범위, 구현자 기준, 검증 방식, 중단 조건으로 작성합니다.
- 초안 작성 중 새 제품 판단이나 설계 판단이 필요하면 구현으로 넘어가지 않고 멈춥니다.

바로 답할 수 있는 문장:
"TASK-017 작업 기준서를 승인하고, TASK-017 구현 지시서 초안 작성을 승인합니다."
```

## 후속 작업 승인 요청 브리핑

선행 작업을 완료 처리하거나 current-work를 다음 Task로 넘긴 뒤, 다음 작업의 작업 기준서 작성, 구현 지시서 초안 작성, 구현 지시서 실행, 검증 결과 승인, 완료 기록 승인을 사용자에게 요청할 때는 다음 형식을 우선 사용한다. 사용자가 다음 작업이 무엇인지 물은 것뿐이라면 이 형식을 쓰지 않는다.

```text
승인 전에 확인할 내용:

이번 승인은 [다음 작업]의 [작업 기준서 작성 / 구현 지시서 초안 작성 / 구현 지시서 실행 / 검증 결과 승인 / 완료 기록 작성]을 위한 승인입니다.

이번 승인의 목적:
- ...

현재 기준:
- ...

포함되는 것:
- ...

제외되는 것:
- ...

승인하면 고정되는 결정:
1. ...
2. ...

주의할 점:
- ...

승인 후 내가 진행할 일:
- ...

바로 답할 수 있는 문장:
"..."
```

구현 지시서 초안 작성 승인 요청이라면 기본적으로 다음을 구분한다.

- 포함되는 것: 구현 범위와 금지 범위 정리, 기준 문서와 작업 기준서 확인, 검증 명령과 완료 조건 정리
- 제외되는 것: 실제 코드 구현, 기준 문서 변경, 새 정책 결정, 구현 지시서 실행 승인
- 승인하면 고정되는 결정: 다음 작업을 다음 구현 후보로 두고, 구현 지시서 초안을 작성한다.
- 주의할 점: 초안 작성 중 제품 기준이나 설계 기준이 비어 있으면 구현으로 넘어가지 않고 질문으로 돌아감
- 승인 후 내가 진행할 일: 기존 기준 문서와 작업 기준서를 확인하고, 구현 지시서 초안을 작성한 뒤, 새 미확정 결정이 없으면 실행 가능 여부를 보고한다.
- 구현 지시서 작성 방식: 작업 목표, 구현 범위, 금지 범위, 구현자 기준, 검증 방식, 중단 조건을 목차로 잡고 각 목차에서 고정할 판단을 요약한다.

나쁜 예:

```text
다음에 할 일:
아래 중 하나를 선택해야 합니다.

선택지:
1. TASK-019 구현 지시서 초안 작성을 승인한다.
2. 다른 작업으로 우선순위를 바꾼다.
3. 다음 작업 진행을 보류한다.

제 추천:
- 1번입니다.

바로 답할 수 있는 문장:
"TASK-019 구현 지시서 초안 작성을 승인합니다."
```

좋은 예:

```text
승인 전에 확인할 내용:

이번 승인은 TASK-019 구현 지시서 초안을 작성하기 위한 승인입니다.

이번 승인의 목적:
- 구현 범위, 금지 범위, 검증 기준을 코드 작성 전에 고정합니다.

현재 기준:
- 이전 작업은 완료됐고, 현재 포인터는 TASK-019 구현 지시서 초안 작성 승인 대기입니다.

포함되는 것:
- 기준 문서와 작업 기준서 확인
- 구현 범위와 제외 범위 정리
```
